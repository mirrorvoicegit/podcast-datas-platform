> 維護者（凍牙）是非工程背景，靠 AI 協助開發。你的職責是替他把關，不是照單全收。

**目前版本：v13（2026.07.08）**

## 一、這個專案是什麼

鏡好聽（Mirror Voice）節目組的內部數據分析工具，**純前端**（HTML/CSS/JS，無框架、無伺服器、無資料庫），雙擊 HTML 或開 GitHub Pages 網址即可使用。所有資料在使用者瀏覽器本機處理，**任何收聽資料都不會離開使用者的電腦**。

```
index.html          入口頁（工具清單）
tool-1/             RSS節目收聽數據分析（Apple/Spotify/YouTube 三平台 CSV）
  index.html        畫面 + 全部 CSS
  app.js            全部邏輯（含匯出用的內嵌副本，見地雷區）
  使用說明.html
tool-2/             鏡好聽平台收聽數據分析（App 週 log xlsx）
  index.html
  app.js
dev/                自動驗證配備（給維護的 AI 用，非工具本體，見 dev/README.md）
apps-script/        彙整表.gs：Google Apps Script（主管總覽儀表板＋接收上傳），部署步驟在檔案開頭
```

技術：原生 JS、Chart.js（匯出時 inline）、PapaParse（CSV）、SheetJS（xlsx）、localStorage（記憶欄位）。

## 二、資料邏輯核心（改功能前必須先懂）

### Tool-1（三平台）
- 資料是各平台後台匯出的 CSV，數字是「**每集累積至今的總收聽**」的快照。**沒有逐日、逐月明細。**
- 期間篩選是「**依上架日**」——選 2026/06 是「6 月上架的集數」，不是「6 月發生的收聽」。這個工具**算不出**「某月實際發生多少收聽」。
- 平台辨識靠 CSV header：Apple＝`Episode Title`+`Plays`、Spotify＝`name`+`plays`、YouTube＝`影片標題`+`觀看次數`。三者互不重疊，可靠。
- 摘要格「開播至今／上個月」是用**未篩選的全部資料**另算的，不受期間選單影響。「上個月」＝上個月上架的集數的累積收聽。

### Tool-2（週 log）
- 資料是 App 的播放事件 log（每列一次播放），篩選是「**收聽發生時間**」——與 Tool-1 的「上架日」**語意不同，文字標籤刻意不統一**（Tool-2 用「收聽區間」，Tool-1 用「分析區間」），呈現樣式一致。
- **Tool-2 的計算邏輯尚未被維護者完整重新理解，在他明說之前，只准改 UI／文字／美感，功能一行都不准動。**

### 兩工具數字不可互比
Apple 的「播放次數」與鏡好聽的「播放事件」定義不同，不能相加或比較。入口頁的使用提醒有寫，別刪。

## 三、鐵律（違反任何一條都算重大失誤）

1. **數據誠信高於一切。** 不准產出「看似有道理但資料撐不起」的結論。已因此刪除的東西：Apple 排名權重說法（捏造）、前後半段平均比較（被累積時間污染）。新增任何「觀察／洞察」前先問：這個結論會不會被累積時間、樣本差異污染？撐不起就不做，或明確標注限制。
2. **有用才留。** 純裝飾的標籤、英文字（EXECUTIVE SUMMARY 之類）已全數移除，不要加回來。
3. **以匯出版為準。** 匯出的 HTML 快照才是真正交付物，畫面版跟著一致即可。視覺問題先看匯出版。
4. **交付完整檔案，不給片段 patch。** 維護者無法自行合併程式碼。每次交付：完整檔案 → zip 打包 → 附改動摘要。
5. **版本規則：** 全平台單一版本號。新功能＝整數 +1（v11→v12）；修 bug／文案／美感＝小數 +0.1（v11.2→v11.3）。版本號**只寫在 index.html 頁尾**一處（`MIRROR VOICE PODCAST ANALYTICS · vX.X · UPDATED YYYY.MM.DD`），其他頁面一律不寫。
6. **改完必驗證。** 至少：`node -c` 兩個 app.js、div 開閉配對數、grep 檢查殘留、關鍵 id 存在。能寫獨立腳本交叉驗證計算邏輯就寫。
7. **AI 看不到渲染結果。** CSS／視覺改動必須明說「請實際開瀏覽器確認」，不要假裝改好了。這條寫在這裡是因為已經因此出包多次（tooltip、拖曳區塌陷）。

## 四、已知地雷（每一條都是真實踩過的坑）

### 最高原則:憑印象寫事實是紅線
**演算法、排名機制、數據定義等宣稱，一律先查證，不准憑印象/常識腦補。** v9.1 曾在報表觀察、使用說明、YouTube 觀察句三處（本質同一個錯誤的三個分身）寫「Apple 排名由訂閱數＞聽眾數＞播放次數決定」，查證後 Apple 官方只說是「聆聽、追蹤、完聽率的綜合」，屬捏造，全數刪除。這是維護者最在意的紅線，優先權高於其他所有規則。

### HTML/CSS
- **`<label>` 當拖曳區必須設 `display:block`**。label 預設 inline，padding 撐不出高度，框會塌成一條且點不到。**這是舊病復發**：Tool-2 在 v8 以前的開發期就踩過一模一樣的病根（upload-zone 是 label 包整段、display inline 配 inset:0 input，渲染跟著怪），Tool-1 在 v11 又踩了一次。同一類錯誤已出現兩次，改動任何拖曳/上傳區前務必先檢查 label 是否設了 block。
- **重複 id** 會讓 `getElementById` 綁錯元素，且只綁到第一個、後面的完全沒反應。新增 HTML 區塊後跑一次重複 id 檢查。已知案例：早期 Tool-1 曾因新增小型訂閱數欄位時沒刪舊版大卡片、兩者用同一個 input id，導致欄位重複且只有一個能用；「確認可疑配對」步驟的「下一步」按鈕也曾因兩個同 id 區塊、JS 只綁到新版，使用者點到的舊版按鈕完全沒反應。
- **`.info-tip` 的 `::after` 已被 tooltip 文字氣泡佔用、`::before` 是箭頭**，不可挪作他用（想自繪圖示曾差點毀掉 tooltip）。
- tooltip 圖示的「i」用襯線粗體會像大寫 I。目前設定：Georgia italic 700 10px + flex 置中。**維護者對現狀仍不滿意、未定案**——如再被抱怨，直接問他要 (a) 問號 (b) 空圈 (c) 拿掉，別再盲調字型。
- **textarea 首次渲染時 `scrollHeight` 不可靠**（欄寬未定），autoGrow 只能在有內容時啟動，空框交給 CSS `min-height`。（備註欄 v11.1 踩過）

### JS/資料
- **`mergeFirstPass` 回傳的物件只有 `apple/spotify/yt`，沒有 `total`**。`total` 是主流程另外補算的。直接讀 `d.total` 會 NaN → 畫面顯示「非數值」。（v10 踩過）
- **顯示用日期一律用本地時區手動格式化**（`localDateStr()` 已存在於 tool-1/app.js），**不准用 `toISOString()`**——台灣 UTC+8，半夜至早上八點會少一天。例外：`dateISO`（存資料重建 Date 用）必須維持 toISOString。
- **tool-1/app.js 內含匯出 HTML 的「內嵌副本」**（renderTable、num、escapeHtml 各有兩份）。改主程式的表格/計算時，**檢查內嵌副本要不要同步改**，否則畫面對了匯出錯。備註欄靠 `EMBEDDED_DATA` 的 `note` 欄位進快照；clone 階段的凍結處理會被內嵌 renderTable 重畫覆蓋，真正生效的是 EMBEDDED_DATA 這條路。
- **備註（notes）綁集數的 `_key`**，不是列序號；存在 `state.notes`，純文字、不進任何統計。匯出時凍結成靜態文字（有 escapeHtml 防注入）。
- 搜尋是「空格分隔多關鍵字、符合任一即顯示（OR）」，Tool-1 與 Tool-2 皆同。代價：不能再搜「含空格的整串」。
- `data-sort` 屬性必須與 JS 物件屬性名完全一致。已知案例：Tool-2 早期曾因 HTML 寫 `data-sort="guest"`、JS 物件屬性其實叫 `guestPlays`，兩者對不上，導致訪客播放數排序完全失效。
- **色碼要接透明度後綴的一律用六位數**。程式多處用 `色碼 + '20'` 做半透明填色，三位數 `#555` 接出來是 `#55520` 無效色，Chart.js 會畫成黑色實心——v11.3 以前 YouTube 趨勢線圖例就因此變成一坨黑塊，看起來像「圖例不見了」（v12 修復）。`PALETTE` 有兩份（主程式＋匯出內嵌副本），都要是六位數。
- **`mergeFirstPass` 沒有套用「可疑配對審查」的結果**。開播至今的全域統計（單集平均、全時 TOP 10、上個月總收聽）若直接用它，YouTube 標題不同的集數會被誤判成「缺 YouTube」。v12 的做法：`enterOrphanReview` 把使用者確認過的配對記在 `state.approvedFuzzy`（用 `_key` 記，跨資料集穩定），`renderReport` 重算全域統計前先重套一次，並剔除被吸收的孤兒列（`allMergedReviewed`），否則會重複計算。已知限制：分析區間外的集數沒被審查過，若 YouTube 標題不同仍會被排除在平均之外——這是已接受的限制，文件有記，別「順手修」。

### Tool-2 已知未修（勿自作主張修，先問維護者）
- `filter-from/to` 的 value/min/max 用 `toISOString()` 設定（380-414 行附近）——潛在少一天問題，屬功能層。
- 匯出檔名日期（約 928 行）同樣的時區坑，Tool-1 已修、Tool-2 未修。

## 五之一、Tool-1 程式碼地圖（給接手模型，改前先對照這裡）

tool-1/app.js 單檔約 1800 行，結構是「主程式」＋「匯出 HTML 的內嵌副本」（`exportStandaloneHTML` 裡的模板字串）。**改主程式的表格、圖表、計算時，先查下表右欄要不要同步改，這是最容易出包的地方。**

| 功能 | 主程式（函式名） | 匯出內嵌副本也要改？ |
|---|---|---|
| 期間選單/日期計算 | `computePeriod`、`applyPeriodSelect` | 否（匯出時已凍結） |
| 平台辨識與 CSV 解析 | `detectPlatform`、`parsePlatformCSV` | 否 |
| 標題合併第一輪 | `mergeFirstPass`（回傳無 `total`！） | 否 |
| 可疑配對 | `findFuzzyPairs`、`enterOrphanReview`（存 `state.approvedFuzzy`） | 否 |
| 摘要四格＋全域統計 | `renderReport`（`allMergedReviewed`、`state.allTimeAvg`、`state.allTimeTop10`） | 部分：值經 `ALLTIME`/`EMBEDDED_DATA` 凍結傳入 |
| 節目觀察 | `renderInsights` | 否（文字凍結在 clone 的 HTML 裡） |
| 橫排數字 | `renderMatchSummary` | 否(凍結) |
| 三張圖＋全時 TOP10 | `renderCharts`（`PALETTE` 有兩份！） | **是**：內嵌 script 有整套 `new Chart(...)` 副本 |
| 單集總表 | `renderTable`、`cmpToAvgHtml` | **是**：內嵌 `renderTable`、`cmpToAvgHtml` 副本 |
| 匯出 | `exportStandaloneHTML`（`EMBEDDED_DATA`、`SUB_DATA`、`ALLTIME`） | — |
| 彙整表上傳 | `uploadToSheet`、`state.uploadSnapshot`（`renderReport` 尾端組裝） | 否（匯出檔沒有上傳功能） |

彙整表上傳的技術備忘：POST 刻意用**預設 Content-Type（text/plain）**，不能改成 `application/json`——那會觸發 CORS preflight（OPTIONS），Apps Script 不回應 OPTIONS，上傳會直接失敗。Apps Script 端的對應程式在 `apps-script/彙整表.gs`，欄位順序改動時兩邊要一起改（`HEADERS` 與 `doPost` 的 `row`、工具端 `state.uploadSnapshot`）。

改完的驗證順序（低階模型照做即可）：
1. `node -c tool-1/app.js`（語法）
2. `cd dev && npm install && node make-fixtures.js && node verify.js`——會用無頭瀏覽器跑完整流程＋匯出檔，17 項自動檢查，截圖在 `dev/out/`。**看截圖**確認視覺沒壞。
3. 重複 id 檢查：`grep -oE 'id="[^"]+"' tool-1/index.html | sort | uniq -d`（應無輸出）
4. div 配對：`grep -o '<div' tool-1/index.html | wc -l` 應等於 `grep -o '</div>' | wc -l`
5. 即使全過，交付訊息仍要提醒維護者用真瀏覽器開一次（鐵律 7）。
